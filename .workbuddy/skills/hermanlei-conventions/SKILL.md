# 厉害了哥的开发约定 (hermanlei-conventions)

本 Skill 定义了 hermanlei（厉害了哥）在日常开发中积累的核心习惯和偏好。
所有协助 hermanlei 的工作都必须遵守这些约定，它们来自真实项目中的血泪教训。

## 适用范围

hermanlei 当前的主要工作方向：

- **Unity 编辑器工具开发** — 自定义 EditorWindow、PropertyDrawer、AssetPostprocessor 等，代表作品为 SceneCraft（场景工坊）工具套件
- **渲染方向** — 图形渲染管线、Shader 开发、材质系统、渲染优化
- **P4 (Perforce) 集成** — 在 Unity 编辑器内集成 P4 版本控制操作

本约定中的 UI 设计偏好、按钮交互模式、颜色编码等规则，均来自上述方向的实际开发经验。
**当协助 hermanlei 开发相关工具时，必须加载并遵守本 Skill。**

## 设计哲学（一切规则的"为什么"）

> **一句话版**：用机制代替人力，用数据驱动代替硬编码，用小步验证代替大步翻车。

这个 Skill 里所有的铁律、规范、流程，都围绕**三个核心信念**：

1. **一个人 + AI 也能做出专业团队的质量** — 前提是用机制（闭环流程、检查清单、自动巡检）替代"有人审"。规则存在不是为了限制你，而是替你盯着那些容易忘的事。

2. **踩过的坑是最贵的知识，必须留住** — 每条铁律背后都是一次真实事故（正则重构炸 7 个文件、P4 没备份回不去…）。踩坑 → 记录 → 变成规则 → 下次不再踩。这就是这个 Skill 的生长方式。

3. **细节决定成败，但不是所有细节都同等重要** — 所以有温度分级（🔴铁律 > 🟡规范 > 🟢知识）、有任务分级（🟢轻量 / 🟡标准 / 🔴重度）。该严的地方寸步不让，该快的地方不拖泥带水。

> _把规则内化为习惯，习惯就不再是负担。_

**温度迁移规则**：
- 🟢→🟡：某个"参考"级别的实践，在 3 次以上的实际开发中被证明有效 → 升为规范

- 🔴→🟡：极少发生，除非项目技术栈彻底变了导致某条铁律不再适用
- 任何层级→删除：技术过时、项目废弃、被更好的替代

### Skill 使用规范（AI 行为准则）

> 不管是哪个 AI 在用这个 Skill，都必须遵守以下行为准则。

1. **加载 Skill 后先报告**：告诉用户"已加载 hermanlei-conventions Skill"，不需要列出所有规则
2. **加载后执行过期检查**：读 MEMORY.md 中的 `last_weekly_check` 和 `last_monthly_research` 时间戳，超过 10 天 / 35 天 → 在回复开头一行提醒用户（详见成长引擎「Skill 加载时的过期触发检查」）
3. **铁律不可跳过**：🔴 铁律级别的规则，即使用户说"不用管那个"也要**提醒一次**（提醒后用户仍坚持则执行）
4. **不确定时查文档**：碰到领域问题先查 references/，不要凭空回答
5. **改 Skill 必须告知**：任何对 SKILL.md 或 references/ 的修改，都要列出改了什么并等用户确认
6. **不要过度加载**：一次最多叠加 2 个 Skill（本 Skill + 1 个领域 Skill），多了 AI 上下文会爆
7. **working memory 是桥梁**：对话中的临时发现写入 working memory → 确认有价值后再迁入 Skill
8. **中断任务必须记录**：当用户切任务/说暂停/会话即将结束时有未完成的工作 → **立刻**把未完成事项写入 MEMORY.md「待恢复任务」区块。批量工作（如分析多个文件）必须**做完一个写一个**，不要攒着最后一起写
9. **诊断/审查 Skill 时的行为规范**：用户问"有什么缺陷/改进"时，必须按以下框架回答，**禁止上来就倒一堆改进清单**：
   - **第一步：先说现有机制的价值**——"这些规则已经防住了什么、哪些设计是成熟的"
   - **第二步：🔴 不做会出事的缺陷**——最多 3 条，每条说清楚"不修的后果是什么"
   - **第三步：🟡🟢 锦上添花的建议**——压缩到 2-3 条，不展开细节，一句话带过
   - **原因**：AI 天生倾向于"找问题"，不加约束会制造焦虑——把"理论上更好"和"不做会出事"混在一起报，让用户误以为 Skill 一身毛病

## 零、工具开发闭环流程（顶层工作流）

> **核心原则**：不是写完代码就结束，而是 设计→搭骨架→填肉→验证→预审→复盘→更新规则 的完整闭环。
> 每一轮闭环的产出（教训、规则、架构文档）都会喂给下一轮，形成正向飞轮。

### 任务分级（先判断再动手）

> **不是所有任务都需要走完整 8 步闭环。** 先花 5 秒判断任务等级，再决定走哪些步骤。

| 等级 | 判断标准 | 走哪些步骤 | 例子 |
|------|----------|-----------|------|
| 🟢 **轻量** | 改动 ≤ 1 个文件 且 ≤ 30 行，无架构变化 | 直接改 → 验证 → 完事 | 修 typo、调参数、改文案、加注释 |
| 🟡 **标准** | 改动 2~5 个文件 或 30~200 行，单模块内 | ①④⑤⑧（跳过 ②③⑥⑦） | 加小功能、修 bug、小规模重构 |
| 🔴 **重度** | 改动 > 5 个文件 或 > 200 行，涉及架构/跨模块 | **全部 8 步，不跳** | 新模块、大重构、新工具、架构变更 |

**分级规则**：
- **拿不准时按高一级处理** — 宁可多走一步也别漏了关键检查
- **过程中发现低估了 → 随时升级** — 本来以为是 🟡 结果越改越多，立即切换到 🔴 流程
- **🔴 铁律不受分级影响** — 不管什么等级，重构铁律（第一章）和备份检查单都必须遵守

### 8 步闭环

```
① 方案设计 → ② 架构评审 → ③ 程序框架 → ④ 增量实施(+L1守门)
                                                    ↓
⑧ 更新规则 ← ⑦ 复盘总结 ← ⑥ 预审检查 ← ⑤ 落地验证
```

| 步骤 | 名称 | 做什么 | 谁主导 | 产出物 | 跳过条件 |
|------|------|--------|--------|--------|----------|
| ① | **方案设计** | 明确做什么、为什么、不做什么。不需要长篇大论，一段话说清楚即可 | 用户 | 需求描述 | 10 行以下小改可跳过 |
| ② | **架构评审** | AI 出架构方案（文件拆分、类职责、数据流），**用户审** | AI 提案→用户拍板 | 架构文档 / 口头确认 | 单文件改动可跳过 |
| ③ | **程序框架** | 搭空壳：类定义、方法签名、接口、TODO 注释，**不写实现**。确保编译通过、结构合理 | AI | 可编译的骨架代码 | 功能 < 50 行可跳过 |
| ④ | **增量实施** | 一个功能一个功能填肉。每填完一块自动跑 L1 日常守门（见 8.9 节） | AI（用户验收） | 可运行的代码 | — |
| ⑤ | **落地验证** | 用户在 Unity 里实际操作，验功能、验手感 | 用户 | Bug 反馈 / 体验意见 | — |
| ⑥ | **预审检查** | 按 8.9 节的 L2/L3 机制做代码预审，对照台账和规则 | AI | 更新台账 | 小改跟 L1 守门合并 |
| ⑦ | **复盘总结** | 这次踩了什么坑？什么决策对了？什么模式以后要避免？ | 用户 + AI | 复盘记录 | 用户说"不用复盘" |
| ⑧ | **更新规则** | 把复盘结论写回 SKILL.md / 台账 / 架构文档 | AI | 规则迭代 | 没有新发现时跳过 |

### 验证哲学：没有团队，靠机制

> 传统团队每个环节有人审（产品审需求、主程审架构、QA 测功能）。
> 我们是"一个人 + AI"，没有第三方审核者。
>
> **解法不是找人审，而是让每个环节的结果"可被验证"：**
> - 方案设计 → 用挑战性问题逼出答案（有答案 = 被验证）
> - 架构评审 → 用自检清单替代架构师（全打勾 = 通过）
> - 程序框架 → 用编译器验证（编译过 = 结构没问题）
> - 增量实施 → 用编译+运行+规则三层叠加（覆盖面 ≥ Code Review）
> - 落地验证 → 用第①步的验收标准逐条对照（前后呼应，不漏项）
> - 复盘总结 → 用对比法替代主观判断（对照台账/规则/历史，客观事实说话）
>
> **核心思想：把"人审"变成"机制审"，把"拍脑袋觉得行"变成"对照清单全打勾"。**

### 各步骤详细说明

> **详细展开见** `references/editor-ui-detailed-rules.md` 第八节。以下只列核心要点。

| 步骤 | 核心要点 |
|------|---------|
| ① 方案设计 | AI 必问 3 个挑战性问题：不做会怎样？有没有更简单方式？怎么知道做对了？ |
| ② 架构评审 | 输出文件列表+职责+依赖方向+数据流。过 7 项自检清单。用户拍板后动手 |
| ③ 程序框架 | 搭空壳（类+方法签名+TODO），必须编译通过 |
| ④ 增量实施 | 一次一个功能，三层验证叠加（编译+用户操作+L1守门） |
| ⑤ 落地验证 | 回溯①的验收标准，逐条对照打勾 |
| ⑥ 预审检查 | 见预审工作流（`references/scenecraft-dev-rules.md` 第六节） |
| ⑦ 复盘总结 | AI 主动问"要复盘吗？"，对照台账+规则+上次做对比 |
| ⑧ 更新规则 | 有新发现 → 写回 SKILL.md/台账/架构文档。没有就跳过 |

### 轻量 vs 重量：按规模选择

不是每次改代码都要跑完 8 步。规模决定深度：

| 改动规模 | 跑哪些步骤 | 举例 |
|----------|-----------|------|
| **微调**（< 10 行） | ④ 直接改 | 改个标签文字、调个颜色 |
| **小功能**（10~100 行） | ①④⑤ | 加个按钮、加个检查规则 |
| **中功能**（100~500 行） | ①②④⑤⑥ | 新增一个检查器面板 |
| **大功能 / 新工具**（500+ 行） | 全部 8 步 | 新建 EditorWindow 工具 |

AI 在收到需求后，先判断规模，告诉用户"这个大概是中功能级别，我会跑 ①②④⑤⑥"。

## 一、代码重构铁律

> **2026-03-27 血的教训**：用 PowerShell 正则脚本自动替换 GUIStyle 变量名 `s`，
> `\b` 匹配短变量名导致正则表达式、字符串字面量、函数参数被全局污染，
> 7+ 个文件严重损坏，被迫 P4 全量回滚。

### 十条铁律（不可违反）

**重构安全（1-6）：**

1. **绝不用正则做 C# 批量重构** — `\b` 匹配短变量名会污染正则表达式、字符串字面量、函数参数。只用手工逐文件替换或 AST 感知工具。
2. **改代码前先确认可回滚** — P4 项目没有 git，改之前必须告诉用户"我要改这些文件"，让用户确认有回滚手段。
3. **改完一个文件立即验证** — 不要攒一堆改动最后才发现炸了。做一个验一个。
4. **出了问题不要硬修** — 超过 2 轮修不好就停下来，告诉用户需要回滚哪些文件，让用户操作。不要越修越烂。
5. **linter ≠ 编译器** — VSCode/OmniSharp 说零错误不等于 Unity 能编译通过。不要以 linter 结果作为"修好了"的依据。
6. **不写 PowerShell/Bash 脚本做代码变换** — 这不是文本替换，是代码变换，需要理解语法树。

**技术交付质量（7-10）：**

> **2026-03-30 教训**：减面工具直接上手写，没查业界成熟方案，交付的工具没法用，用户时间白费。

7. **涉及技术深度的工具/功能，先查参考再写代码** — 不是所有东西都需要查（改个 UI 文案不用），但凡涉及算法、数据结构、图形学、性能优化等有技术门槛的领域，必须先花 15-30 分钟搜集业界成熟方案（引擎源码、GDC/SIGGRAPH 分享、3A 游戏技术文章、成熟开源库），建立参考基线后再动手。
8. **方案要有出处，不靠"我觉得"** — 给用户的技术方案，关键决策点必须能说出"谁用过、在什么项目里验证过"。如果找不到权威出处，要明确告诉用户"这部分是我的推断，没有找到业界先例"。
9. **最小可用版本先交，确认能跑再迭代** — 不要一口气写完整个工具再交付。先交核心功能最小版本，用户确认能跑、方向没问题，再加功能。减少"写了一大堆发现方向错了"的浪费。
10. **超出能力范围就说，建议用现成库** — 如果某个需求涉及的技术深度超出 AI 可靠输出的范围（如复杂的半边数据结构、数值求解器等），直接告诉用户"这个建议用现成库（如 XXX），我来做集成"，不硬写容易出 bug 的实现。

### 重构前备份检查单（每次重构开始前必须执行）

开始任何涉及多文件修改的重构前，按以下顺序执行：

1. 列出即将修改的所有文件清单
2. 提醒用户："这些文件即将被修改，请确认已有备份/回滚手段（P4 shelve、.bak 文件、或其他方式）"
3. 等用户明确确认后再动手
4. 如果是 P4 项目，提醒用户先 Checkout 相关文件

未经用户确认，**禁止**开始修改文件。

### 安全重构四件套（修老代码时必须选用）

> **背景**：修老代码比写新代码危险十倍。业界三大流派（Feathers 特征测试、Fowler 绞杀者模式、2025-2026 AI 辅助重构）
> 的核心共识是：**不是靠"小心点"来避免出错，而是靠机制让出错时能被立刻发现并回滚。**

根据改动规模，从以下四个工具中选择组合使用：

#### 工具一：不变量声明（所有改动必须执行）

开始任何重构前，AI 必须先输出不变量声明：

```
## 不变量声明
- 要改的：[具体说明改什么、怎么改]
- 不能变的：
  - 界面外观（按钮位置、颜色、文字不变）
  - 功能逻辑（输入输出行为不变）
  - 公共接口（方法签名、参数类型不变）
  - [其他具体约束]
```

用户看过声明后，改完可以针对性验证"不能变的"有没有变。
**没有不变量声明就开始改代码 = 违规。**

#### 工具二：改前快照（中等以上改动执行）

> 来源：Michael Feathers "特征测试 / 黄金主控" 的简化版，适配无测试框架环境。

在改代码之前：
1. **操作快照**：在 Unity 里把当前功能跑一遍，请用户记录关键输出（截图 / Console 日志 / 结果数量）
2. **代码快照**：把原文件复制一份 `.bak`（P4 项目用 shelve）

改完之后，同样操作一遍，跟快照对比：
- 输出跟改前一样 = 没改坏
- 输出不一样 = 要么是有意改的（bug 修复），要么是无意改坏的

> 关键：**快照记录"现在是什么"，不管对不对。先锁定行为，再谈改善。**

#### 工具三：绞杀者模式（God Class / 大规模重构时使用）

> 来源：Martin Fowler 2004 年提出，2025-2026 仍是业界改造遗留系统的主流策略。

**不在老代码里动刀，在旁边写新的替代它。** 流程：

1. **新建替代文件**：在旁边写 `XXX_V2.cs`（或更好的命名），实现同样的功能
2. **逐个迁移调用**：老代码里的调用逐个改成调用新文件
3. **并行验证**：新旧代码并存期间，两边都能跑，随时可以切回旧代码
4. **删除旧代码**：所有调用都迁移完毕、验证通过后，删除旧文件

好处：
- 旧代码始终完整可回滚（因为你没改它）
- 新代码从一开始就是干净的架构
- 如果新代码出问题，把调用改回旧的就行

**适用场景**：P4SubmitChecker（6874 行）、SceneAssetChecker（5995 行）等 God Class 的拆分。

#### 工具四：单点手术（小 bug 修复 / 小改动时使用）

**只改一个方法，不顺手改旁边的。** 严格遵守：

1. 一次只改一个方法体内的代码
2. 不改方法签名（除非这就是本次要改的东西）
3. 改完立刻编译 + 测试这一个功能
4. 确认没问题后再改下一个
5. **每改 3~5 个方法就请用户在 Unity 里编译验证一次**

> "顺手改"是万恶之源。你去修一个 GUIStyle 缓存，顺手改了旁边的变量名，
> 结果变量名改错了，但你以为是 GUIStyle 缓存导致的 bug，越修越远。

### 按改动规模选择组合

| 改动规模 | 必选 | 推荐 |
|----------|------|------|
| 修一个 bug / 改一行 | 不变量声明 + 单点手术 | — |
| 缓存 GUIStyle / 改 5~10 处 | 不变量声明 + 单点手术 | 改前快照 |
| 提取方法 / 拆分文件 | 不变量声明 + 改前快照 | 绞杀者模式 |
| God Class 拆分 / 大规模重构 | 不变量声明 + 改前快照 + 绞杀者模式 | — |

### 老代码消化策略（系统性变好，不是靠"顺手"）

> **核心问题**：台账记了问题，安全重构四件套防翻车，但"什么时候修、按什么顺序、每次修多少"没有定义。
> 没有策略的"顺手消化"= 失控的连锁修改 = 又翻车。

#### 两条路径：被动消化 vs 主动消化

| 路径 | 触发条件 | 做什么 | 安全边界 |
|------|----------|--------|----------|
| **被动消化** | 因新功能/bug 修复**必须**碰某个文件时 | 顺手修掉该文件里的台账问题 | 本次顺手修的量 ≤ 本次功能改动量 |
| **主动消化** | 用户专门拿出时间说"今天消化技术债" | 按优先级框架选择目标，专门修 | 一次只修一个台账条目 |

#### 被动消化的安全边界（"顺手"的尺度）

**铁律：顺手改的量不能超过正事改的量。**

举例：
- 你改 `SceneAssetChecker.cs` 加一个新检查规则（正事，改了 30 行）→ 可以顺手缓存该文件里 2~3 个 GUIStyle（顺手，改 10 行）
- ❌ 你改一行 bug fix → 顺手重构 200 行 → **这不叫顺手，这叫借题发挥，禁止**

具体规则：
1. **只修同文件**——不要因为顺手而跨文件改
2. **只修台账里已记录的问题**——不要修着修着发现新问题也一起改
3. **顺手改完也要过不变量声明**——确认没影响到不相关的功能
4. **超出安全边界时 AI 必须刹车**：告诉用户"这个顺手修的量太大了，建议单独开一轮主动消化"

#### 主动消化的节奏

用户说"今天消化技术债"时：
1. AI 读台账，按优先级框架（见下文）推荐 1~3 个条目
2. 用户选一个
3. 走安全重构四件套（不变量声明 → 改前快照 → 改 → 验证）
4. 改完更新台账状态（⬜ → ✅）
5. **一个条目完全搞定后**，再问用户"要继续下一个吗？"

> 关键：**一次只开一个手术台**。不要同时修两个台账条目。

### 技术债优先级框架（Debt Triage）

> **用途**：台账里几十个问题，不可能全修，也不应该全修。这个框架决定"先修什么、后修什么、永远不修什么"。

#### 四级优先级

| 级别 | 标准 | 什么时候修 | 台账里的典型例子 |
|------|------|-----------|----------------|
| **P0 — 定时炸弹** | 会导致崩溃、数据丢失、死锁、安全漏洞 | **尽快修**，下次碰到这个文件必须修 | SYS-01 Process 死锁（缓冲区满永久阻塞） |
| **P1 — 慢性毒药** | 不会崩溃但持续制造痛苦：重复代码导致改一处漏一处、God Class 导致每次改都可能误伤 | **主动消化时优先选**，被动消化时如果顺手量合适也修 | SYS-04 三套重复 P4 实现、SYS-03 God Class |
| **P2 — 性能债** | 不影响正确性，但影响体验：卡顿、GC 抖动、每帧分配 | **被动消化时顺手修**，不值得专门花时间 | SYS-02 OnGUI new GUIStyle 62 处 |
| **P3 — 美容债** | 命名不规范、注释缺失、格式不统一 | **永远不专门修**，碰到了才顺手改 | 变量命名不符合 7.3、缺注释 |

#### 优先级判定规则

AI 在台账里给新问题定级时，按以下顺序判断：
1. **会不会崩？** 会 → P0
2. **会不会导致连锁修改失败？**（改一处必须同步改 N 处，漏了就出 bug）会 → P1
3. **用户能感知到性能问题吗？** 能 → P2
4. **以上都不是？** → P3

#### 什么时候算"修完了"

台账条目从 ⬜ 改为 ✅ 的标准：
- 代码已改 + 编译通过 + 用户在 Unity 里验证功能正常
- 如果是 P0/P1：必须走完改前快照 + 改后对比
- 如果是 P2/P3：编译通过即可

### God Class 拆分操作手册（绞杀者模式实操）

> **适用对象**：单文件超过 3000 行的巨型类（SceneCraft 的 P4SubmitChecker 6874 行、SceneAssetChecker 5995 行等）。
> **核心理念**：不在 God Class 里动刀，在旁边长出新代码，逐步替代。

#### Step 0：评估是否值得拆

不是所有 God Class 都需要拆。问三个问题：
1. **这个类还在频繁新增功能吗？** 不再新增 → 不拆，留着用
2. **每次改它都容易出 bug 吗？** 不容易 → 不拆
3. **多人（含 AI）需要同时修它吗？** 不需要 → 不拆

> 三个都是"否" → **不拆**。God Class 不好看，但"能用且不常改"就别碰它。

#### Step 1：识别可独立的"器官"

读代码（或读架构速查文档），找出 God Class 里**职责独立的方法群**：

```
例：P4SubmitChecker 6874 行
├── P4 连接/登录逻辑（~800 行）→ 可独立为 P4ConnectionManager.cs
├── 文件差异分析逻辑（~1200 行）→ 可独立为 P4DiffAnalyzer.cs
├── 提交检查规则（~2000 行）→ 可独立为 P4SubmitRules.cs
├── GUI 绘制（~2500 行）→ 留在原类（它是 EditorWindow）
└── 工具方法（~300 行）→ 归入 P4AssetHelper.cs
```

**判断标准**：这组方法只访问自己的数据，不跟其他方法群共享字段 → 可独立。

#### Step 2：一次只摘一个"器官"

**绝不同时拆两个。** 流程：

1. 选一个最独立的方法群（依赖最少的）
2. 新建文件（`P4ConnectionManager.cs`），把方法原封不动复制过去
3. 新文件里调通编译（可能需要调整访问级别、传参）
4. **旧文件里这些方法先不删**——改成调用新文件的对应方法：
   ```csharp
   // 旧：P4SubmitChecker.cs 里的方法
   private bool P4Login(string password)
   {
       // 以前 200 行的实现...
   }

   // 改成转发：
   private bool P4Login(string password)
   {
       return _connectionManager.Login(password);  // 转发到新类
   }
   ```
5. 编译 + 用户验证功能不变
6. 确认 OK 后，**下一轮**再考虑删掉旧方法里的转发，让调用方直接用新类

#### Step 3：验证"器官移植"成功

每摘一个器官后：
- [ ] 新文件编译通过
- [ ] 旧文件编译通过（转发到新文件）
- [ ] 用户在 Unity 里操作相关功能，行为不变
- [ ] 台账更新：标记 God Class 拆分进度

#### Step 4：重复直到 God Class 瘦到合理体量

目标不是"完全消灭 God Class"，而是瘦到 **< 1500 行**。
- EditorWindow 类保留 GUI 绘制逻辑（这本来就该在这里）
- 业务逻辑、数据处理、外部进程调用全部在独立文件里

#### 拆分安全红线

- ❌ **不改公共接口签名**——其他文件调用的方法，签名保持不变，内部改成转发
- ❌ **不改已有功能的行为**——拆分是结构变化，不是逻辑变化
- ❌ **不在拆分的同时修 bug**——一次只做一件事
- ❌ **不一次提交所有拆分**——每摘一个器官，编译验证一次，确认 OK 后再下一个

## 三、功能模块间接口设计

### 共享工具类

- 多个编辑器工具共用的功能（如 `GetP4ExePath()`）抽到共享工具类（如 `P4Helper`）
- 共享方法修改时，**所有调用方必须同步更新**（如 `P4AssetHelper` 和 `P4SubmitChecker` 同步更新 `GetP4ExePath()`）
- 共享类以 `static` 方法或单例提供，不要每个工具各自实现一份

### 模块间通信

- 模块间**不直接引用彼此的实例**，通过以下方式解耦：
  - **EditorPrefs**：用统一的 `PREFS_PREFIX` 存取配置（如 `"PrefabMaster_P4Port"`）
  - **共享工具类**：公共方法作为桥接
  - **事件/回调**：需要实时通知时用 `System.Action` 回调，不用轮询
- EditorPrefs key 命名规范：`{PREFS_PREFIX}{模块名}_{配置项}`

### 向后兼容

- `PREFS_PREFIX` 和 `TRACEABILITY_PREFIX` 保持旧前缀（如 `"PrefabMaster_"`），即使工具已改名
- 改名时不迁移旧配置，直接兼容读取

## 五、P4 (Perforce) 工作流

### 操作规范

- P4 项目**没有 git 的撤销能力**，所有修改前必须确认回滚手段
- 批量修改前提醒用户先 `p4 edit`（Checkout）相关文件
- 使用 `p4 -s` (tagged output) 解析命令输出，比纯文本解析更可靠
- P4CLIENT 环境变量经常是错误的（机器名默认值），工具中需要主动检测和修复
- 工具内嵌 `p4.exe` 时，目录名不能用 `~` 后缀（如 `Bin~`），Unity 不生成 .meta 会导致 P4 trigger 报错

### Unity + P4 集成注意

- P4V 的 workspace 配置和 `p4` 命令行用的 P4CLIENT 可能不一致
- 通过 `p4 clients -u 用户名` 列出 workspace，按 Root 路径匹配工程
- 密码登录通过 `Process.StandardInput.WriteLine(password)` 喂给 `p4 login`
- **🔴 GUI 进程（Unity/WinForms 等）调 p4 必须包 chcp 65001** — Windows 默认 Console Code Page 是 936 (GBK)，p4.exe 在 GBK 下对含中文路径的文件 digest 计算错误，导致 reconcile 返回大量假阳性。统一用 `cmd /c "chcp 65001 >nul 2>&1 && p4 args..."` 包装，同时设 `StandardOutputEncoding = Encoding.UTF8`。详见 wiki pitfalls.md P4-2

#### Skill 自我成长规则

> Skill 不只靠用户主动喂养——AI 必须在日常对话中**主动发现**值得沉淀的内容并提议更新。

**主动提议触发条件**（满足任一即触发）：

1. **新最佳实践**：AI 在回答中给出了重要的规范/模式/技巧，但当前 Skill 里没有记录
2. **工作流变更**：发现用户的项目结构、工具链、开发习惯发生了变化（新项目、新路径、新工具）
3. **重复问题**：同一类问题在不同会话中被问到第二次——说明值得写成规则
4. **新反模式/踩坑**：发现了一个新的坑或反模式，值得加入铁律
5. **路由表缺口**：某个领域被频繁触发但没有专用 Skill，说明该补了
6. **规则过时**：发现 Skill 中某条规则已不适用当前项目状态（路径变了、工具换了、流程改了）

**提议格式**：
```
💡 建议更新 Skill：[具体建议内容和理由]
```

**执行规则**：
- 提议放在回答末尾，**不打断当前工作流**
- 用户确认后才执行更新，**不得自动修改 Skill**
- 一次最多提 2 条建议，避免信息过载
- 琐碎/临时性的内容不提议（只沉淀有长期价值的）

#### Skill 成长引擎（三个定期机制）

> Skill 不是一次性写完的文档，是一个**活的知识系统**。以下三个机制确保它持续进化。

##### 机制一：每周巡检（健康检查）— 自动化任务，每周一 14:30

> 维护现有内容的准确性。

**巡检内容**：
1. 路由表中「待补充」的领域，搜索是否有新的合适 Skill 可用
2. 已安装 Skill 列表与路由表是否一致（有没有装了但没加到表里的）
3. Skill 中引用的文件路径是否仍然有效
4. 最近一周的 working memory 日志中，是否有反复出现但未被 Skill 覆盖的模式
5. 是否有规则因项目变化而需要更新
6. **错过补救检查**：检查上次巡检日期（记录在 working memory 的 MEMORY.md 中），如果距今 >10 天说明中间错过了巡检，自动补上
7. **内容退化/过时检测**（🆕）：
   - 检查每个项目/领域在最近 30 天 working memory 中是否被提及。**超过 60 天未提及的领域 → 标记为"休眠"**
   - 休眠领域的 SKILL.md 专属章节和 references/ 文件，在巡检报告中列出并建议用户："XXX 领域已 60 天未活跃，是否需要精简/归档相关内容？"
   - 检查 references/ 中每个文件的"最后更新"日期，**超过 90 天未更新的文档 → 标记为"可能过时"**
   - 巡检分拆信号表（第 4 条）：检查是否有某个领域的专属内容超过总量 40%
8. **知识库记忆演化**（🆕 — 配合 knowledge-guardian 第五章）：
   - 逐条扫描 `unity-knowledge-guardian/references/pitfalls.md` 和 `bug-tracker.md`
   - 对每条判断：Unity/引擎版本升级是否已修复？技术栈变更是否导致不再适用？创建 > 180 天且无触发记录？
   - 输出建议列表 → 用户确认后才标注 `[已过期:原因]`（绝不自动标注）
   - `bug-tracker.md` 中 `[已修复]` 超 90 天的移入「历史归档」区段
   - 更新 pitfalls.md 快速索引表的"活跃/过期"计数

**MEMORY.md 中需要维护的活跃度追踪**（AI 每次在某领域做了实质工作后更新）：
```markdown
## 领域活跃度
- scenecraft: 2026-03-30
- biosphere-game: 2026-03-28
- rendering: 2026-03-30
- hermanlei-conventions: 2026-03-30
```
> 格式：`领域名: 最近一次实质工作日期`。巡检时用这个判断是否休眠。

**输出**：简短报告 → 用户确认后执行。

**错过补救机制**：
> 自动化任务依赖 WorkBuddy 运行。如果用户某天没打开 WorkBuddy，任务就不会执行。

| 场景 | 补救方式 |
|------|----------|
| 错过了某周的巡检 | 下次打开时，AI 在首次对话开始前检查 MEMORY.md 中的 `last_weekly_check` 日期，超过 10 天自动触发补巡检 |
| 错过了某月的前沿调研 | 下次打开时，AI 检查 `last_monthly_research` 日期，超过 35 天自动触发补调研 |
| 长期不使用（>60 天） | AI 不要一口气补所有错过的，只做一次"回归巡检"：检查 Skill 中所有路径是否有效 + 做一次前沿调研 |

**Skill 加载时的过期触发检查**（方案 B — 自动化任务的兜底）：
> 自动化任务是主力，但如果 WorkBuddy 那天没运行，自动化就不会执行。
> 以下规则让 **每次加载 hermanlei-conventions Skill** 时都做一次快速过期检查，确保不会长期错过。

**AI 每次加载本 Skill 后，在开始正式工作前，花 3 秒执行以下检查**：
1. 读取当前 workspace 的 `MEMORY.md`（或 hermanlei-conventions Skill 目录下的 `.workbuddy/memory/MEMORY.md`）
2. 检查 `last_weekly_check` 距今是否 > 10 天
3. 检查 `last_monthly_research` 距今是否 > 35 天
4. 如果任一过期 → 在回复开头用一行提醒用户：
   ```
   ⚠️ 巡检/调研过期提醒：上次巡检距今 N 天（> 10 天），上次调研距今 M 天（> 35 天）。要补一次吗？
   ```
5. 用户确认后执行补巡检/补调研；用户说"不用"就跳过
6. 如果 MEMORY.md 不存在或没有时间戳字段 → 提醒用户"巡检系统还未初始化，要初始化吗？"

**不做的情况**：
- 当前任务是紧急 bug 修复 → 不打断，任务完成后再提醒
- 当前会话已经在执行巡检/调研 → 不重复提醒

**MEMORY.md 中需要维护的时间戳**（AI 每次执行巡检/调研后更新）：
```markdown
## Skill 成长引擎状态
- last_weekly_check: 2026-03-30
- last_monthly_research: 2026-03-30
```

**领域调研矩阵**：

**调研流程（每个领域通用）**：
1. AI 搜索该领域 2~3 个关键词
2. 筛选出**对当前项目有实际价值**的发现（不是所有新东西都值得关注）
3. 输出调研报告，每个发现标注等级：
   - ⭐ **值得采纳**：成本低、收益高、与当前项目直接相关
   - 👀 **值得关注**：有潜力但暂不行动
   - ℹ️ **了解即可**：纯学术/实验阶段或与当前场景无关
4. 用户确认后：
   - ⭐ 级 → 写入对应知识索引或 SKILL.md
   - 如果是新领域且没有知识索引 → 按 `references/README.md` 规范创建新文档
5. **追加到 `unity-knowledge-guardian/references/research-index.md`**（调研成果总索引），一行包含：编号 R-NNN、日期、领域、主题、核心结论、存放位置、关键词

**调研成果的流向规则**（新发现该写到哪里？）：

```
新发现
  ├── 是具体技术知识（如"Unity 6.5 新增了 XX API"）→ 对应领域的知识索引
  ├── 是好的编程实践（如"发现一种更好的单例替代"）→ SKILL.md 编码规范章节
  ├── 是新的工具/Skill（如"发现一个好用的 Unity Skill"）→ 路由表
  ├── 是一个新的坑/反模式（如"Unity 6 的 XX 功能有 bug"）→ 对应项目的已知问题台账
  └── 不确定归哪 → 先写 working memory，下次巡检时决定
```

**评估标准**（决定一个新技术/实践是否值得采纳）：
1. **跟我们的场景相关吗？** 不相关 → 跳过
2. **实施成本 vs 收益？** 成本高于收益 → 只标 👀 不采纳
3. **成熟度够吗？** 纯学术/实验 → 标 ℹ️，等它成熟再评估
4. **会不会影响现有架构？** 影响太大 → 谨慎，除非收益巨大

##### 机制三（附加）：成长回顾（学了有没有用上）— 随月度调研一起执行

> 光学不用等于没学。每次月度调研时，**先回顾上个月学到的东西落地了没**。

**回顾流程**（5 分钟，调研之前先做）：
1. **翻上次调研报告**中标 ⭐ 的采纳项 → 逐条检查：实际用上了吗？
2. 分三种结果：
   - ✅ **已落地** — 在哪用的？效果怎样？→ 写一句话总结到调研报告的"回顾"段
   - ⏳ **没机会用** — 还值得保留吗？→ 保留到下个月再看，连续 2 个月没用 → 降级为 👀
   - ❌ **试了不行** — 为什么不行？→ 记录原因，避免下次又采纳同类东西
3. **输出一行总结**："上月 N 个采纳项，M 个落地，K 个待验证，J 个淘汰"

**记录位置**：写在当月调研报告的开头，格式：
```
## 上月回顾（YYYY-MM）
- ✅ XXX — 在 YYY 中使用，效果良好
- ⏳ XXX — 暂未有使用场景，保留观察
- ❌ XXX — 原因：...
> 总结：上月 3 个采纳项，2 个落地，1 个待验证
```

**竞品/同类游戏调研**（游戏领域特有）：
> 当用户在做游戏项目时（当前：biosphere-game），调研不只看技术，也看**同类游戏的设计和表现**：
- 同类游戏的创新玩法和机制
- 同平台（微信小游戏 / 移动端）的技术限制和最佳实践
- 用户体验和界面设计趋势

##### 机制三：沟通模式沉淀（经验提炼）— 每次对话结束时

> 把用户的思维方式和决策模式沉淀下来。

**用户的沟通思路模式**（基于历史对话总结）：

| 模式 | 表现 | AI 应该做的 |
|------|------|-------------|
| **由点到面** | 从一个具体问题出发，拓展到"全球有没有更好的做法" | 不止解决眼前问题，同时做横向调研对比 |
| **先防守后进攻** | 先确保不翻车（安全重构），再追求变好（消化策略） | 回答时先给防护方案，再给优化方案 |
| **实用主义** | "好不好"不看理论多漂亮，看能不能落地、成本高不高 | 每个建议都标注实施成本和收益 |
| **体系化思维** | 不满足于零散的规则，要把规则组织成可检索的体系 | 新内容加入时考虑它在 Skill 中的位置，不要随便塞 |
| **成长导向** | 不只是"现在能用"，更关心"以后怎么越来越好" | 每次重大改动后回头检查 Skill 是否需要更新 |
| **元思维** | 不只关心"做什么"，更关心"怎么让做事的方式本身也在进步" | Skill 自身也需要规范和迭代，不只是内容在长，结构也要优化 |

> AI 在每次对话结束时自问：**这次对话中有没有新的决策模式或偏好值得沉淀？** 有 → 更新此表 + working memory。

##### 机制四：Skill 自身的版本迭代规范 🆕

> Skill 本身也是"代码"，也需要规范来管理它的成长。

**Skill 修改的分级制度**：

- SKILL.md 主文件控制在 **1500 行以内**——超过了说明有内容该迁移到 references/
- 单个 references 文档控制在 **300 行以内**——超过了说明该拆分

## 七、通用 C# 编码规范（所有项目适用）

> **不管是 SceneCraft 编辑器工具、小游戏、渲染模块还是任何 C# 项目，都必须遵守这些规则。**

### 7.1 不要重复自己（DRY）

- **写新功能前，先检查项目中有没有现成方法可以复用**
- 如果两处代码做的事情超过 80% 相似，必须抽取为共享方法
- 共享方法放在**专门的工具类/Helper 类**中，不要散落在各个业务类里

### 7.2 方法和类的体量控制

- **单个方法** ≤ 50 行（超了就拆成子方法，方法名即注释）
- **单个文件** 新写的尽量 ≤ 500 行（历史遗留的大文件容忍，但新文件别再这么大）
- 如果一个方法需要滚动才能看全——它太长了
- 如果一个类能用"和"来描述它的职责——它该拆了

### 7.3 命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 私有字段 | `_camelCase` | `_playerHealth`, `_syncFileList` |
| 公有属性 | `PascalCase` | `Instance`, `Value`, `MaxHealth` |
| 方法 | `PascalCase` | `Initialize()`, `CalculateDamage()` |
| 常量 | `UPPER_SNAKE_CASE` | `MAX_RETRY_COUNT`, `DEFAULT_SPEED` |
| 局部变量 | `camelCase` | `filePath`, `hitCount` |
| 布尔变量 | `is/has/can/should` 前缀 | `_isReady`, `hasPermission` |
| 事件回调 | `On` 前缀 | `OnEnable()`, `OnDamageReceived()` |
| 接口 | `I` 前缀 | `IDamageable`, `ISaveable` |

### 7.4 修改现有代码的规矩

1. **改共享类/工具类时，必须检查所有调用方是否需要同步更新**
2. **改方法签名时，全局搜索该方法名**，确认没有遗漏的调用
3. **删除代码前确认没有其他文件引用**
4. **改完一个文件立即验证**——不要攒一堆改动最后才发现炸了

### 7.5 错误处理 & 防御式编程

**外部进程调用：**
- 所有外部进程调用必须设 **超时**（推荐 30 秒），超时后 Kill 进程并给用户明确提示
- 用 `Process.Start()` 时必须重定向 `StandardOutput` + `StandardError`，两者都要读
- 进程退出码非 0 时，把 stderr 内容展示给用户，不要默默吞掉

**文件和路径：**
- 操作文件前先 `File.Exists()` / `Directory.Exists()` 检查
- 路径拼接用 `Path.Combine()`，不要手拼 `"/" + xxx`
- 从用户输入获取的路径要 `Trim()` 并处理末尾多余的 `/` 或 `\`
- 文件读写包在 `try-catch` 里，catch 中给用户人话提示（不是堆栈信息）

**外部进程调用示例：**

```csharp
// ❌ 错误 — 没超时、没读 stderr、没检查退出码
var p = Process.Start("p4", "reconcile -n ...");
p.WaitForExit();
string output = p.StandardOutput.ReadToEnd();

// ✅ 正确 — 超时 + 重定向 + 退出码检查
var psi = new ProcessStartInfo("p4", "reconcile -n ...")
{
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false,
    CreateNoWindow = true
};
using var proc = Process.Start(psi);
string stdout = proc.StandardOutput.ReadToEnd();
string stderr = proc.StandardError.ReadToEnd();
if (!proc.WaitForExit(30000)) // 30秒超时
{
    proc.Kill();
    Debug.LogError("外部进程超时，已强制终止");
    return;
}
if (proc.ExitCode != 0)
    Debug.LogWarning($"进程退出码 {proc.ExitCode}: {stderr}");
```

**try-catch 原则：**
- **不要大范围 try-catch 吞异常** — 只包裹可能出错的具体操作（IO、进程、网络）
- catch 中必须做两件事：① 给用户看得懂的错误提示 ② `Debug.LogException(e)` 保留完整堆栈
- **绝不** catch 后空处理（`catch { }`）

**try-catch 示例：**

```csharp
// ❌ 错误 — 大范围 try-catch 吞掉所有异常
try
{
    LoadConfig();
    ParseData();
    UpdateUI();
    SaveResults();
}
catch (Exception e) { Debug.LogError(e.Message); }

// ✅ 正确 — 只包裹可能出错的具体 IO 操作
LoadConfig();
ParseData();

try { SaveResults(); }  // 只有文件 IO 需要 try-catch
catch (IOException e)
{
    EditorUtility.DisplayDialog("保存失败", $"文件写入出错：{e.Message}", "确定");
    Debug.LogException(e);  // 保留完整堆栈
}
// 验证：catch 块里必须同时有 ① 用户提示 ② LogException

UpdateUI();
```

### 7.6 注释 & 代码文档规范

**必须写注释的地方：**
- 复杂算法 / 业务逻辑前加注释说明**为什么这样做**（不是做了什么——代码本身说明做了什么）
- Hack / Workaround 必须加 `// HACK:` 或 `// WORKAROUND:` 前缀 + 原因说明
- 魔法数字必须有注释或提取为 `const`
- 公共 API 写 `/// <summary>` XML 注释

**注释语言：**
- **用中文**。这是给 hermanlei 和中文团队看的项目
- XML Summary（`/// <summary>`）也用中文，方便 IDE 悬停提示

**TODO / FIXME 标记：**
- `// TODO:` — 计划要做但还没做的
- `// FIXME:` — 已知有问题需要修的
- `// HACK:` — 临时方案，以后要改的
- 所有标记必须附上**简短说明**，不能光写 `// TODO:`

### 7.7 Unity 通用规范（游戏 + 工具都适用）

> 以下规则适用于所有 Unity C# 代码，不限于编辑器工具。

- **`GetComponent` 结果缓存** — 不要在 Update/OnGUI 中每帧调用 `GetComponent<>()`，在 Awake/OnEnable 中缓存

```csharp
// ❌ 错误 — 每帧调用 GetComponent
void Update()
{
    var rb = GetComponent<Rigidbody>();  // 每帧 GC + 查找开销
    rb.AddForce(Vector3.up);
}

// ✅ 正确 — Awake 中缓存
private Rigidbody _rb;
void Awake() => _rb = GetComponent<Rigidbody>();
void Update() => _rb.AddForce(Vector3.up);
```

- **字符串比较用 const** — Tag、Layer、Animator 参数等不要用魔法字符串，提取为 `const` 或 `static readonly`
- **`SerializeField` 优于 public** — 需要在 Inspector 暴露的字段用 `[SerializeField] private`，不要直接 public
- **`EditorUtility.SetDirty()` 不能忘** — 通过脚本修改 ScriptableObject/Asset 后必须调，否则改动不保存
- **避免 `Update()` 里做可以事件驱动的逻辑** — 能用事件/回调/协程的，不要每帧轮询
- **`null` 检查注意 Unity 假 null** — Unity 重写了 `==` 运算符，`Destroy` 后的对象 `== null` 为 true 但 `is null` 为 false。统一用 `== null` 而非 `is null`

```csharp
// ❌ 错误 — C# 原生 null 检查，被 Destroy 后的对象骗过
if (target is null) return;               // Destroy 后仍返回 false！
if (ReferenceEquals(target, null)) return; // 同上

// ✅ 正确 — Unity 重写的 == 运算符能正确识别 Destroyed 对象
if (target == null) return;               // Destroy 后正确返回 true
```

| ❌ 不要用 | ✅ 统一用 | 原因 |
|-----------|----------|------|
| `obj is null` | `obj == null` | `is null` 绕过 Unity 的 `==` 重写 |
| `ReferenceEquals(obj, null)` | `obj == null` | 同上，无法检测 Destroyed 状态 |
| `obj is not null` | `obj != null` | 同理 |

- **协程异常不会冒泡** — `StartCoroutine` 中的异常只会停止协程，不会抛到外层。关键协程要自己 try-catch

### 7.8 通用踩坑记录

> 所有项目通用的坑。每次踩坑追加到这里。

| # | 踩坑场景 | 后果 | 正确做法 |
|---|---------|------|---------|
| 1 | 用 PowerShell 正则批量替换 C# 代码 | 文件污染，全量回滚 | 手工逐文件替换，绝不用正则脚本 |
| 2 | PowerShell 中 commit message 含特殊字符 | git commit 失败或乱码 | 用 `-F` 文件方式传递，UTF-8 编码 |

| 4 | Unity 假 null（`is null` vs `== null`） | 空引用异常或逻辑错误 | Unity 对象统一用 `== null` |
| 5 | 直接写有技术深度的工具（如减面工具），没查业界方案 | 交付的工具没法用，用户时间白费 | 先查参考建立基线再写代码（铁律第 7 条） |

| 8 | `Lightmapping.lightingSettings` 在 null 时读取 | 抛 Exception（不是返回 null） | 必须 try-catch 包装：`try { return Lightmapping.lightingSettings; } catch { return null; }` |

### 7.9 游戏开发特有规范（biosphere-game 等）

> **温度**：🟢 知识级 — 这是初始框架，待实际踩坑后逐步升温为 🟡 规范。
> **详细知识索引**：`references/gamedev-knowledge-index.md`

#### 架构原则（SO 优先 + 组合模式）

1. **共享状态用 ScriptableObject，不用单例/静态字段** — 生命值、分数、配置等运行时共享数据，创建 `FloatVariable`/`IntVariable` 等 SO 资产
2. **跨系统通信用 SO 事件通道，不用直接引用** — `GameEvent : ScriptableObject` + `GameEventListener`，禁止 `GameObject.Find()` / `FindObjectOfType()`
3. **每个 MonoBehaviour 只做一件事** — 能用"和"描述职责的组件必须拆分
4. **预制件必须自包含** — 拖入空场景无报错、无对场景层级的隐式依赖
5. **用 `[SerializeField] private` 暴露字段** — 不要 public 字段满天飞
6. **数据驱动优于硬编码** — 数值、配置、敌人属性等全部走 SO 或配置文件，设计师可在 Inspector 直接调整

#### 性能预算意识

- **Update() 里的逻辑要有理由** — 能事件驱动的不轮询，能协程的不每帧
- **GetComponent 必须缓存** — Awake/OnEnable 中缓存，不在热路径中调用
- **对象池（Object Pool）** — 频繁创建/销毁的对象（子弹、特效、敌人）必须用池
- **移动端帧预算** — 微信小游戏/移动端目标 30fps，每帧预算 ~33ms
