# 编辑器 UI 详细规则 & 闭环流程详细步骤

> **用途**：Unity 编辑器工具 UI 设计的详细参考（颜色编码、排版、交互模式）和闭环流程各步骤的展开说明。核心原则在 SKILL.md 第二章和第零章，本文档是按需查阅的详细补充。
> **类型**：📋 开发细则
> **维护规则**：新增 UI 规则或发现更好的交互模式时追加
> **最后更新**：2026-03-30
> **保鲜周期**：项目活跃时实时更新

---

## 一、UI 排版布局规则

| 规则 | 说明 |
|------|------|
| **Toolbar 固定顶部** | 工具栏（按钮行、颜色图例条）固定在窗口/区域顶部，不随 ScrollView 滚动消失 |
| **功能区用 HelpBox 包裹** | 逻辑上独立的功能块用 `EditorStyles.helpBox` 背景包裹，视觉分隔清晰 |
| **Foldout 分组折叠** | 多个功能模块使用 Foldout 折叠分组，默认展开核心功能、折叠次要功能 |
| **列表区用 ScrollView** | 文件列表、检测结果等可变长内容必须包在 `EditorGUILayout.BeginScrollView` 中 |
| **状态信息内联显示** | 连接状态、用户名等关键信息直接显示在相关控件旁（如 "✅ P4 已连接 (用户名@workspace名)"），不要单独弹窗 |
| **页签分离大功能** | 功能差异大的模块用页签（Tab）分离（如"提交预检" vs "版本管理"），不要挤在一个长页面 |
| **输入框+下拉预设并排** | 可编辑输入框右侧紧贴 ▼ 下拉按钮，点击展开预设列表（如服务器地址预设） |

## 二、颜色编码系统

预览列表等场景中的颜色图例使用 **■ 方块 + 实际颜色 + 文字说明** 的格式：

| 颜色 | 用途 | 备注 |
|------|------|------|
| **美术场景色**（自定义） | `scene_making/scenes` 下的美术场景文件 | ART_FOCUS_FOLDERS 只限核心场景目录 |
| **绿色** | 其他美术资源（非场景的美术目录文件） | ART_FOCUS_FOLDERS 缩小后，其余美术用绿色 |
| **冲突色**（自定义） | 有冲突的文件 | |
| **上次跳过色**（自定义） | 上次同步时被跳过的文件 | |
| **紫色** `(0.7, 0.5, 1.0)` | 需关 Unity 才能同步的文件/文件夹 | ❗ 不用灰色，避免和普通灰色混淆 |
| **非美术新增色**（自定义） | 非美术目录的新增文件 | |
| **黄色**（文字色） | 非场景目录含场景子文件时的提示文字 | 如 "含N个场景更新" |
| **绿色**（高亮背景） | Workspace 选择中自动匹配的 workspace | 绿色高亮 + 一键按钮 |
| **灰色** | Workspace 选择中不匹配的 workspace | 灰色显示但允许强制使用 |

**颜色调整历史教训**：
- 需关 Unity 的项最初用灰色，和普通灰色混淆 → 改为紫色
- 图例最初用纯文字描述 → 改为 ■ 方块 + 实际颜色，直观

## 三、分步引导 UI（Step Wizard）模式

适用于需要多步骤初始化/连接的场景（如 P4 连接引导）：

1. **每步独立状态** — 每个 Step 有自己的检测状态变量（如 `_p4ExeFound`、`_p4EnvScanned`、`_p4LoginState`）
2. **初始状态用特殊值** — 用 `-2` 表示"从未检测"，区别于 `0`（检测失败）和 `1`（检测成功），避免 OnGUI 误触发
3. **步骤间有依赖** — Step 2 依赖 Step 1 通过，Step 3 依赖 Step 2 通过，未通过的步骤不展示后续内容
4. **每步有独立按钮+结果显示** — 检测按钮 + 检测结果（成功/失败 + 详细信息）+ 修复操作按钮
5. **检测结果要缓存** — 如 `_p4EnvScanned` 防止每帧重复执行耗时操作
6. **重置时全链路同步** — 重置某步状态时，必须同步重置其依赖的缓存标记（如 `_p4LoginState = 0` 时同步重置 `_p4EnvScanned`）

## 四、列表/树形展示规则

| 规则 | 说明 |
|------|------|
| **颜色图例固定顶部** | 图例条在 ScrollView 外面，不随列表滚动 |
| **子节点继承父节点样式** | 树形结构中子节点继承父节点颜色标记（通过 `parentIsArt` 等参数传递） |
| **工具自身文件自动过滤** | 列表中自动过滤掉工具自己的文件（如过滤 `/SceneCraft/` 路径的条目） |
| **非叶子节点含特定子项加提示** | 非场景目录如果包含场景子文件，加黄色 "含N个场景更新" 提示 |
| **ART_FOCUS_FOLDERS 要精确** | 焦点目录只限核心目录（如 `scene_making/scenes`），不要范围过大 |
| **需关 Unity 的文件夹列表可扩展** | `Settings`、`scripts_dll` 等需关 Unity 同步的目录应维护在可配置列表中 |

## 五、连接/登录流程设计

**零配置理念**：用户打开工具应该尽可能不需要手动输入任何配置。

| 阶段 | 设计 |
|------|------|
| **探测可执行文件** | 多级回退：缓存 → EditorPrefs → 内嵌文件 → PATH 环境变量 → 同类进程探测 → 常见安装路径 |
| **读取环境配置** | 自动从 `p4 set` 命令输出 + Windows 注册表读取配置值，缺失时自动补全 |
| **默认值填充** | 无配置时，服务器地址自动填公司主服务器（如 `192.168.31.15:1667`），不留空 |
| **输入框可编辑** | 自动探测到的值显示在**可编辑**输入框中，允许用户手动修改 |
| **密码登录** | Unity 内直接弹密码输入框，通过 `Process.StandardInput.WriteLine(password)` 喂给 `p4 login`，不需要开命令行 |
| **回车确认** | 密码输入框支持回车键直接提交 |
| **ticket 过期** | 自动弹 `EditorInputDialog` 密码框，支持回车确认，至少 3 次重试机会 |
| **完全零配置** | 提供服务器地址 + 用户名 + 密码三合一界面，一键"配置并登录" |

## 六、按钮交互模式

| 模式 | 说明 | 示例 |
|------|------|------|
| **一键排除** | 一个按钮同时排除多种危险/不可操作项 | 「仅安全」同时排除冲突文件 + 需关 Unity 文件夹 |
| **内联下拉** | 状态行旁附带下拉按钮，不跳转页面 | 已连接状态行的「切换分支」下拉、P4PORT 的 ▼ 服务器预设 |
| **状态行内联信息** | 连接成功后在同一行显示关键上下文 | "✅ P4 已连接 (用户名@workspace名)" |
| **首次操作加说明** | 用户从未执行过的操作按钮附加详细说明文字 | 首次检测 P4 连接按钮下方的说明文字 |
| **耗时操作加等待提示** | 执行后需等待的按钮旁标注提示 | 预览更新按钮旁 "点击后请耐心等待" |
| **一键自动修复** | 检测到问题时提供一键修复，不让用户手动执行命令 | Workspace 不匹配 → 扫描 + 一键 ApplyWorkspace |
| **匹配高亮 + 强制可用** | 自动匹配项绿色高亮 + 一键确认；不匹配项灰色但仍可强制使用 | Workspace 选择列表 |

## 七、Workspace 选择 UI

- 显示 **stream 分支名**和**描述信息**，支持多分支开发场景
- 自动匹配的 workspace 绿色高亮 + 一键应用按钮
- 不匹配的灰色显示，但允许强制使用（别拦死用户）
- 状态行加「切换分支」下拉按钮和「刷新」按钮

## 八、闭环流程各步骤详细说明

> 以下是 SKILL.md 第零章 8 步闭环中各步骤的展开说明。日常只需看 SKILL.md 的简表即可，需要深入理解某步骤时来这里查阅。

### ① 方案设计

用户不需要写正式文档，口头描述即可。AI 的职责是**追问关键缺失**：
- 这个功能给谁用？（自己 / 美术 / 策划）
- 预期操作多大数据量？（影响架构选型）
- 有没有现有代码可以复用？（避免重复造轮子）
- 有没有**不要做**的事？（边界比需求更重要）

**验证锚点 — 3 个挑战性问题**（AI 必须问，用户回答后方案才算"过审"）：
1. **这个功能不做会怎样？**（验证必要性——能不做就不做）
2. **有没有更简单的方式达到同样效果？**（验证是否过度设计）
3. **做完之后，你怎么知道它是对的？**（前置定义验收标准——后面第⑤步靠这个验）

> 有答案的问题就是被验证过的问题。不需要别人审，自问自答也比不问强。

**方案规范模板**（不需要写长文档，但至少把这 5 条填了）：

```
【做什么】一句话说清功能目标
【给谁用】自己 / 美术 / 策划 / 所有人
【不做什么】明确的边界（比"做什么"更重要）
【验收标准】做完后怎么知道它是对的（第⑤步回来逐条对照）
【已知风险】可能踩的坑、性能瓶颈、与现有代码的冲突
```

> **这份规范是活文档**——整个开发过程中保持可访问。第④步实施时发现边界变了，回来更新它。第⑤步验收时拿它逐条打勾。不是写完就扔的一次性文档。

### ② 架构评审

**触发条件**：新工具 / 新模块 / 预计超过 200 行的功能
**AI 必须输出**：
1. 文件列表 + 每个文件的职责（一句话）
2. 类之间的依赖方向（谁引用谁）
3. 数据流：数据从哪来、存在哪、谁读谁写
4. 与现有代码的边界（复用什么、不碰什么）

**关键原则**：
- 单文件不超过 500 行（EditorWindow 工具类）
- 单类单职责，用"和"能描述的就该拆
- P4 操作统一走 P4AssetHelper，不新建第 N 套
- 新建 ScriptableObject 时遵循 `[CreateAssetMenu]` 惯例
- **AI 可读性**：文件名、类名、方法名要让 AI 仅读签名就能理解职责，不依赖内部实现细节。公共方法加一行 `///` XML 注释概述意图

**用户拍板后才动手**。哪怕用户只说"行"一个字，也算确认。

**验证锚点 — 架构自检清单**（AI 出完方案后自己过一遍，把结果展示给用户）：
```
□ 每个文件的职责能用一句话说清吗？（说不清 = 职责不清，拆）
□ 有没有文件会超过 500 行？（超了 = 要拆 partial 或独立文件）
□ 类之间有没有循环依赖？（A→B→A = 架构有环，必须打断）
□ 新增代码有没有和现有代码功能重复？（查台账 + 架构文档）
□ 如果删掉其中一个文件，其他文件能编译过吗？（不能 = 耦合太紧）
□ 非程序员能不能通过 Inspector / 配置文件使用这个功能？
□ AI 能否仅读文件名+类签名就理解模块职责？（不能 = 命名/结构需优化）
```
> 用户不需要看架构本身，只看清单结果。6 个框全打勾 = 架构没大问题。

### ③ 程序框架

先搭骨架再填肉，好处：
- 编译错误在结构阶段就能发现，不用写了 300 行才发现设计不对
- 用户可以看到"全貌"再决定要不要调整
- 方法签名定下来后，多个方法可以并行实施

**骨架标准**：
- 所有类、方法、字段的定义完整
- 方法体只有 `// TODO: 实现 XXX` 或 `throw new NotImplementedException()`
- 必须能编译通过

**验证锚点**：骨架写完后，请用户在 Unity 里编译一次。编译通过 = 结构没问题。这是最便宜的验证——发现问题只需改接口，不需要改实现。

### ④ 增量实施

**核心纪律**：
- 一次只做一个功能点，做完告诉用户"XXX 做好了"
- 每完成一个功能自动跑 L1 守门检查（见预审工作流）
- 发现问题就地修，不攒到后面
- 超出原定架构的需求，先跟用户说"这个超出了原来的设计，要不要调整架构？"

**验证锚点 — 三层叠加**（每完成一个功能块都要过）：
1. **编译通过** — AI 明确说"请编译试一下"，语法错误在这里拦住
2. **用户操作一次** — 功能对不对，用户最清楚
3. **L1 守门自查** — AI 对照规则检查，规范问题在这里拦住

> 三层叠加的覆盖面不比 Code Review 差——Code Review 也看不出运行时 bug。

### ⑤ 落地验证

AI 没法替用户在 Unity 里操作，但可以：
- 提醒用户"建议测这几个场景"
- 用户报 bug 时，优先看是不是架构问题而非实现 bug

**验证锚点 — 回溯验收标准**：
- 拿出第①步定义的"怎么知道它是对的"，逐条对照
- AI 输出一份简短的验收检查清单（基于①的回答生成），用户逐项打勾
- 全部打勾 = 功能验收通过；有没打勾的 = 记录为 bug 回到第④步

### ⑥ 预审检查

具体流程见 `references/scenecraft-dev-rules.md` 的「代码预审工作流」。
关键：不是每次都全量审，台账是缓存，用台账省 token。

### ⑦ 复盘总结

**每个工具 / 大功能做完后，AI 主动问**："要花 2 分钟复盘一下吗？"

复盘模板：
```
## 复盘：[工具名/功能名]（日期）
- ✅ 做对了什么：
- ❌ 踩了什么坑：
- 💡 下次怎么避免：
- 📏 要不要更新规则：
```

用户说"不用"就跳过，但 AI 必须问。

**验证锚点 — 对比法**（避免"你觉得好我也觉得好"的确认偏见）：
- **对照台账**：这次做的东西，有没有犯台账里记录过的**老毛病**？
- **对照规则**：SKILL.md 里的每条相关规则，这次有没有违反的？
- **对照上次**：跟上一个同类功能/工具比，这次哪里更好、哪里更差？

> 不靠主观判断，靠跟历史记录客观对照。数据不会骗人。

### ⑧ 更新规则

复盘有新发现时，AI 把结论写进：
- **SKILL.md** — 新增/修改具体规则条目
- **台账** — 新增已知问题
- **架构文档** — 更新架构描述
- **踩坑记录** — 追加到踩坑表

**不做的事**：不为了更新而更新。没有新发现就跳过这步。
