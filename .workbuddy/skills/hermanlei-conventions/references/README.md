# references/ 目录管理规范

> **最后更新**：2026-04-11

本目录存放 Skill 的**知识层（🟢）内容**——调研报告、技术索引、架构文档、问题台账等。
这些文档不是"规则"（规则在 SKILL.md 主文件里），而是"工具箱"——AI 在需要时查阅。

---

## 一、文档分类

references 目录下的文档分为四种类型，每种有不同的管理规则：

| 类型 | 命名规则 | 示例 | 更新频率 | 谁来更新 |
|------|----------|------|----------|----------|
| **📖 知识索引** | `{领域}-knowledge-index.md` | `rendering-knowledge-index.md` | 每月前沿调研 | AI（用户确认后） |
| **🏗️ 架构速查** | `{项目名}-architecture.md` | `scenecraft-architecture.md` | 架构变更时 | AI（代码变更后自动） |
| **🐛 问题台账** | `{项目名}-known-issues.md` | `scenecraft-known-issues.md` | 发现/修复问题时 | AI（实时更新） |
| **📋 项目上下文** | `project-context.md` | `project-context.md` | 项目变化时 | AI（用户确认后） |

### 什么时候创建新的知识索引？

满足**以下任一条件**时，创建新的 `{领域}-knowledge-index.md`：
1. 用户开始一个新方向的工作（如开始做游戏、做 AI 集成），且对该方向做了全球调研
2. 某个领域的知识量超过 SKILL.md 中一个小节能容纳的（>30 行）
3. 该领域会被**反复查阅**，独立文档比嵌入主文件更高效

### 什么内容不应该放在 references/?

- ❌ **铁律和规范** → 放 SKILL.md（因为 AI 必须主动遵守，不能靠"查阅"）
- ❌ **临时笔记** → 放 working memory 日志
- ❌ **代码文件** → 放项目仓库
- ❌ **单次对话的调研结果** → 先放 working memory，确认有长期价值后再迁移到 references

## 二、文档头模板

每个 references 文档**必须包含**以下头部信息，以便任何 AI 快速理解用途：

```markdown
# [文档标题]

> **用途**：[一句话说明什么时候读这个文档]
> **类型**：📖 知识索引 / 🏗️ 架构速查 / 🐛 问题台账 / 📋 项目上下文
> **维护规则**：[说明谁在什么情况下更新此文档]
> **最后更新**：[YYYY-MM-DD]
> **保鲜周期**：[多久检查一次是否过时]
```

## 三、知识索引的统一结构

所有 `*-knowledge-index.md` 建议遵循此结构（可按需增减）：

```
一、当前状态概览（现在是什么情况）
二、前沿技术图谱（全球最新的是什么，分梯队）
三、与我们的关联（哪些跟当前项目相关）
四、学习资源和信息源（以后去哪查）
五、知识保鲜策略（怎么保持这个文档不过时）
```

---

## 四、文档生命周期

```
调研/开发 → 写入 working memory 日志（临时）
         → 确认有长期价值 → 创建/更新 references 文档（持久）
         → 文档过时 → 标注 ⚠️ OUTDATED 并注明原因
         → 完全无用 → 删除并从 SKILL.md 参考文档章节移除引用
```

**过时判断标准**：
1. 文档中引用的技术/版本已被替代（如 Unity 大版本变更导致 API 失效）
2. 文档中的结论已被新的调研推翻
3. 对应的项目/功能已废弃
