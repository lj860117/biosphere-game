# SceneCraft 开发细则

> **用途**：开发 SceneCraft（场景工坊）编辑器工具时查阅的详细规则和检查清单。铁律和核心规范在 SKILL.md 第八章，本文档是补充细节。
> **类型**：📋 开发细则
> **维护规则**：新增 SceneCraft 开发规则或踩坑时追加；改完代码后检查是否需要更新
> **最后更新**：2026-03-30
> **保鲜周期**：项目活跃时实时更新

---

## 一、体量限制（补充通用 7.2）

- **单个 partial 文件** ≤ 2000 行（超了考虑拆分新 partial）
- **独立工具文件** ≤ 3000 行为宜（已有的大文件是历史遗留，新文件别再这么大）

---

## 二、编辑器 UI 错误展示

- 非致命错误用 `EditorUtility.DisplayDialog()` 弹窗，标题写清什么操作失败了
- 致命错误（初始化失败）在窗口区域用红色大字显示，不要只打 Console.log
- 错误重试不限次数——只有用户点「取消」才退出

---

## 三、#region 组织规则

- 每个 partial 文件按功能用 region 分区，region 名用中文
- region 嵌套不超过 2 层
- 同类方法放同一个 region（所有 Draw 方法放一起，所有数据处理方法放一起）
- **新增 #region 要与现有 region 风格一致**——看同文件中其他 region 怎么命名的

---

## 四、修改代码的额外检查

在通用 7.4 的基础上，额外注意：
1. **第一步永远是读速查文档**（SKILL.md 第八章铁律），通过文档定位文件，不要遍历源码
2. **改共享类（P4AssetHelper / SceneCraftStyles / SceneCraftData / SceneCraftSettings）时，必须检查所有调用方**
3. **partial class 之间的字段引用**——删除代码前确认没有其他 partial 文件在用这个字段

---

## 五、新功能 Checklist

每次实现新功能前，过一遍这个清单：

- [ ] **第一步：读了架构速查文档**（SKILL.md 第八章铁律），通过文档定位到具体文件
- [ ] 查了共享类（SKILL.md 8.1 表格），确认没有可复用的现成方法
- [ ] 新增代码放对了文件（SKILL.md 8.2 表格）
- [ ] 新增配置项走了 `SceneCraftSettings.cs`
- [ ] 新增颜色/样式走了 `SceneCraftStyles.cs`
- [ ] 新增数据类型走了 `SceneCraftData.cs`
- [ ] 方法没超 50 行（通用 7.2）
- [ ] 命名符合通用 7.3 规范

## 六、代码预审工作流

> **目的**：在专业程序员 Code Review 前主动发现问题，同时避免每次都全量审计浪费 token。

### 三层预审机制

| 层次 | 名称 | 触发时机 | 范围 | Token 消耗 |
|------|------|----------|------|-----------|
| L1 | **日常守门** | 每次 AI 改完代码提交前 | 仅本次修改的文件 | 极低（已在上下文中） |
| L2 | **模块预审** | 用户要对某模块做大改之前 | 目标模块 1~3 个文件 | 中等 |
| L3 | **全量预审** | 里程碑节点 / 即将被 Code Review | 全代码库 | 高 |

### L1 日常守门（每次改代码自动执行）

AI 改完代码后，在提交前自动检查本次修改是否触犯以下规则：
1. 是否引入了新的 `new GUIStyle()` 在 OnGUI 路径中？
2. 方法是否超过 50 行？文件是否超过体量限制？
3. 是否使用了同步 `ReadToEnd()` 模式？
4. 是否引入了 `GameObject.Find` / `FindObjectOfType` / 静态单例？
5. 是否有空 catch？是否有硬编码 magic number？
6. 命名是否符合 7.3 规范？

**不需要额外读文件**，因为修改的代码已在上下文中，零 token 额外开销。
**发现问题就地修**，不要等用户指出。

### L2 模块预审（大改之前执行）

用户说"我要重构 XXX"或"给 XXX 加个大功能"时：
1. 先读已知问题台账 `scenecraft-known-issues.md` 中该文件的相关条目
2. 告诉用户"这个文件有 N 个已知问题，要不要顺便修？"
3. 用户确认后，把要修的已知问题和新功能一起做
4. 完成后更新台账（标 ✅）

**核心省 token 策略**：不需要重新读源码找问题，台账里已经有了。

### L3 全量预审（里程碑时手动触发）

用户说"做一次全面预审"或"马上要被 Review 了"时：
1. 读台账，看还有哪些 ⬜ 未修
2. **只对新增的、台账中没有的文件**做全量逐行审计
3. 已有台账记录的文件，只需抽查台账中标 ✅ 的问题是否真的修好了
4. 更新台账

### 已知问题台账维护规则

- **位置**：`scenecraft-known-issues.md`
- **何时读**：L2/L3 预审时、改代码前（查目标文件有无已知问题）
- **何时写**：
  - 修好一个问题 → 改 ⬜ 为 ✅，加修复日期
  - 发现新问题 → 追加到对应文件的章节末尾
  - 新增文件首次审计 → 新增章节
- **快速统计**：台账顶部有汇总表，每次更新后同步刷新数字
- **不做的事**：不把台账当"每次必读的长文"——AI 只需读目标文件对应的章节，不需要全部读完

## 七、SceneCraft 专属踩坑记录

> 编辑器工具开发特有的坑，与通用坑（SKILL.md 7.8）分开记录。

| # | 踩坑场景 | 后果 | 正确做法 |
|---|---------|------|---------|
| 1 | `OnGUI()` 中执行 P4 命令 / 外部进程 | 编辑器卡死 | 必须由按钮点击触发 |
| 2 | GUIStyle 在域重载（Domain Reload）后丢失 | 样式变成空白 | 用 `SceneCraftStyles` 集中缓存 + null 检查 lazy 初始化 |
| 3 | `Bin~` 目录被 P4 trigger 拦截报错 | P4 add 失败 | 过滤带 `~` 后缀的 Unity 临时目录 |
| 4 | P4CLIENT 环境变量默认是机器名 | P4 命令执行在错误的 workspace | 显式传 `-c` 参数或从 p4 set 读取 |
| 5 | 列表"预览 0 文件"问题 | 用户体验差 + 五轮调试 | 显示前验证数据源是否已加载，未加载显示"加载中..." |

> **遇到新坑时，必须追加到此表。这是活文档。**
