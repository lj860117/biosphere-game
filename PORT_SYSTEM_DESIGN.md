# 🔌 端口系统设计方案 v1.0

> **目标**：通过为每种建筑引入可见的端口（Port）机制，将当前"随便放、随意连"的建筑布局升级为"有约束、有策略"的空间规划。
>
> **设计支柱**：
> 1. **一眼可读** — 端口数量和状态在格子上直观可见，无需鼠标悬停
> 2. **即时约束** — 端口数 = 可连接传送带上限，连满即止
> 3. **分类辨识** — 不同建筑角色（源头/转化/增益/旁路/奇观）通过端口形态和颜色一目了然
> 4. **渐进揭示** — P1阶段不显示端口（自动传送带，保持新手友好），P2起显示

---

## 一、核心概念

### 1.1 什么是端口

端口是建筑边缘的**可视化连接槽位**，代表该建筑可以建立的传送带连接数量。

```
           ┌─────────────────┐
  输入端口  │                 │  输出端口
   ●───────→│    🔋 ATP合成酶  │───────→ ●
           │    Lv.2         │───────→ ○
           └─────────────────┘
           1个输入端口(已连)    2个输出端口(1已连/1空闲)
```

- **输入端口（In Port）**：位于建筑**左侧**，表示"能从几条传送带接收资源"
- **输出端口（Out Port）**：位于建筑**右侧**，表示"能向几条传送带输出资源"
- 每个端口一次最多连接**1条传送带**
- 端口数量由建筑类型固定决定，不可动态增减（除非通过科技/突变解锁额外端口）

### 1.2 端口 ≠ 资源种类

当前实现中端口数 = 消耗/产出的资源种类数（如ATP合成酶消耗葡萄糖=1个输入端口）。新方案中端口数 = **可连接的传送带条数**，与资源种类解耦。

| 维度 | 旧方案 | 新方案 |
|------|--------|--------|
| 端口含义 | 1端口 = 1种资源 | 1端口 = 1条传送带连接槽 |
| ATP合成酶 | 1 in (🟢), 1 out (⚡) | 2 in, 3 out |
| 生物膜反应器 | 3 in (🟢🔵⚡), 2 out (🧱🧪) | 3 in, 2 out |
| 端口颜色 | 按资源类型着色 | 空闲=灰色, 已连=资源色, 满=金色 |

### 1.3 端口与传送带容量的关系

端口数限制的是**连接数量**，传送带等级限制的是**每条连接的吞吐量**。两者正交：

```
约束链：端口数 × 传送带等级容量(cap/s) = 该建筑的总吞吐上限

例：ATP合成酶(3 out) + 3条Lv.3传送带(5/s)
   → 最大输出吞吐 = 3 × 5 = 15 单位/s
   → 实际产出 2.5⚡/s × 升级乘数，远低于上限 → 无瓶颈

例：碳源采集器(2 out) + 2条Lv.1传送带(2/s)
   → 最大输出吞吐 = 2 × 2 = 4 单位/s
   → 实际产出 1.5🟢/s → 有余量
   → 但如果升级到Lv.5(乘数2.6x)，产出3.9/s → 接近上限，需升级传送带
```

## 二、建筑端口数值总表

### 2.1 设计原则

1. **源头建筑** — 无输入，输出受限（供给有限，必须选择给谁）
2. **标准转化器** — 输入端口 = 消耗资源种类数，输出端口 = 产出种类数 + 0~1 额外分发口
3. **高效/旁路转化器** — 输入端口适当放宽（鼓励多来源汇聚），输出受限（高效的代价）
4. **增益建筑** — 0 输入 / 0 输出（不参与传送带，靠邻接生效）
5. **奇观** — 仅输出，端口数最多（终极节点，大量分发）
6. **帝国核心供能建筑** — 输出受核心供给上限，端口数体现这一限制

```
  总端口 max = 5 (生物膜反应器 3+2, 共振培养箱 2+3)
```

### 2.4 设计理由：为什么这些数字

#### 碳源采集器：0 in / 2 out — 供给瓶颈的源头

碳源采集器是唯一的源头建筑，受帝国核心供能上限限制（P1:2台→P5:10台）。2个输出口意味着：
- 每台采集器最多给2个下游建筑供葡萄糖
- P1阶段2台采集器 × 2口 = 最多4条输出线
- 玩家必须决定：给ATP合成酶还是简易提取器？给能量缓冲池还是氨基酸合成仪？
- **策略深度**：不能"一台采全场"，必须规划供应链拓扑

#### ATP合成酶：2 in / 3 out — 能量中枢的分发能力

ATP是全游戏最紧张的资源（20种建筑中15种消耗），合成酶需要足够的分发能力：
- 2个输入口：可以从2台碳源采集器接收葡萄糖（或1采集器+1共振培养箱）
- 3个输出口：能同时供给3台下游消费者
- **平衡考量**：如果只给2个输出口，中后期能量瓶颈会更严重；3个口在P1就让玩家感到"够用但不富裕"

#### 旁路建筑统一 1 out — 效率折扣的体感化

旁路建筑（能量缓冲池、氨基酸合成仪、生物质转化炉、噬菌体裂解器、信息素广播站）的共同特征是"提供替代路线但效率略低"。限制为1个输出口：
- 玩家能感知到旁路"好用但有瓶颈"
- 鼓励旁路作为补充而非主力
- 例外：能量缓冲池给2个输出口，因为它的角色更接近"备用能量源"而非"旁路转化"

#### 微型戴森球：0 in / 4 out — 奇观的终极奖励感

奇观是300秒建造的终极目标，4个输出口是对投入的回报：
- 产出12⚡+8🟢+2🧱，需要足够的端口分发
- 4口 × Lv.5传送带(12/s) = 48 单位/s 理论吞吐
- 让奇观"看起来就很强"——边缘四个方向都有管线辐射出去

## 三、端口UI视觉规格

### 3.1 端口形态设计

当前格子尺寸 80×80px（手机约60×60px），端口需要在这个空间内清晰可见。

#### 桌面端规格

```
端口尺寸:    14×14px（原10×10px → 放大40%）
端口间距:    3px
端口形状:
  输入端口 → 半圆形（右半，开口朝建筑内部）
             border-radius: 0 50% 50% 0
  输出端口 → 三角/箭头形（朝右，指向外部）
             clip-path: polygon(0 0, 100% 50%, 0 100%)
  增益建筑 → 无端口，四角发光标记
  奇观     → 圆形端口（比普通大1.5x → 21×21px）
端口位置:
  输入端口 → 左侧边框外延2px（从建筑内部"伸出"左边缘）
  输出端口 → 右侧边框外延2px
  垂直排列 → 居中对齐
```

#### 手机端规格

```
端口尺寸:    10×10px（原7×7px → 放大43%）
端口间距:    2px
最大可显示:  4个端口/侧（(60 - 4*2) / (10+2) ≈ 4.3）
```

#### 端口布局计算

```
端口区域总高度 = 端口数 × (端口高度 + 间距) - 间距
示例：3个输入端口 = 3 × (14 + 3) - 3 = 48px → 80px格子高度内居中，上下各余16px ✓
示例：4个输出端口 = 4 × (14 + 3) - 3 = 65px → 80px格子内，上下各余7.5px ✓（极限）

手机: 4个端口 = 4 × (10 + 2) - 2 = 46px → 60px格子，上下各余7px ✓（勉强可行）
```

### 3.2 端口颜色状态

端口有三种视觉状态，通过颜色和填充传达连接信息：

```
状态        | 填充   | 边框     | 含义
------------|--------|---------|-----------------------------------
空闲 idle   | 透明   | 灰色50% | 可以连接，等待传送带
已连 linked | 资源色 | 资源色  | 已有传送带连接，显示传输的资源颜色
满载 full   | 金色   | 金色    | 建筑所有端口都已连接（整体状态标记）
```

CSS 实现参考：

```css

.port {

  width: 14px;
  height: 14px;
  transition: all 0.2s ease;
  z-index: 8;

/* 全满状态 — 建筑整体发光 */
.cell.ports-full {

}
.cell.ports-full .port.linked {
  border-color: #facc15;
  box-shadow: 0 0 4px #facc15;
}

/* 空闲端口呼吸动画 — 提示可连接 */

}
```

### 3.3 端口内图标

每个端口内部显示该端口关联的资源图标，增强可读性：

```
端口类型   | 图标显示规则
-----------|-------------------------------------------
输入端口   | 显示对应消耗资源的emoji（🟢/⚡/🔵/🧪/🧱/📡）
输出端口   | 显示对应产出资源的emoji
多端口同资源| 显示数字标记：如ATP合成酶3个输出口都是⚡
空闲端口   | 显示虚线圆圈 ○ (无图标)
```

端口与资源的映射规则：

```javascript
// 端口分配逻辑
function getPortConfig(bldType) {
  const bd = BLDS[bldType];
  const config = PORT_DEFS[bldType]; // { maxIn, maxOut }

  // 输入端口分配
  const inPorts = [];
  const consKeys = Object.keys(bd.cons || {});
  for (let i = 0; i < config.maxIn; i++) {
    inPorts.push({
      resource: consKeys[i % consKeys.length] || null,
      // 超出资源种类数的端口为"通用输入口"
      isGeneric: i >= consKeys.length,
    });
  }

  // 输出端口分配
  const outPorts = [];
  const prodKeys = Object.keys(bd.prod || {});
  for (let i = 0; i < config.maxOut; i++) {
    outPorts.push({
      resource: prodKeys[i % prodKeys.length] || null,
      isGeneric: i >= prodKeys.length,
    });
  }

  return { inPorts, outPorts };
}
```

### 3.4 移除/降级的视觉元素

端口成为一等公民后，需要为其腾出视觉空间。建议调整：

## 四、传送带连接约束规则

### 4.1 新增连接前检查

当前 `computeBelts()` 和手动连接逻辑中**无连接数量限制**。新方案增加以下检查：

```javascript
// 新增：端口配置表
const PORT_DEFS = {
  // Phase 1
```

### 4.2 连接数计算函数

```javascript
/**
 * 获取建筑当前已使用的端口数
 * @param {number} idx - 格子索引
 * @param {string} direction - 'in' | 'out'
 * @returns {number} 已使用端口数
 */
function getUsedPorts(idx, direction) {
  const belts = G._activeBelts || [];
  const type = G.grid[idx]?.type;
  if (!type) return 0;

  let count = 0;
  for (const belt of belts) {
    if (belt.fi !== idx && belt.ti !== idx) continue;
    const otherIdx = belt.fi === idx ? belt.ti : belt.fi;
    const otherType = G.grid[otherIdx]?.type;
    if (!otherType) continue;

    if (direction === 'in') {
      // 本建筑是接收方：对端→本端方向匹配
      if (FLOW_MAP.some(f => f.from === otherType && f.to === type)) count++;
    } else {
      // 本建筑是输出方：本端→对端方向匹配
      if (FLOW_MAP.some(f => f.from === type && f.to === otherType)) count++;
    }
  }
  return count;
}

/**
 * 检查建筑是否还有空闲端口可连接
 * @param {number} idx - 格子索引
 * @param {string} direction - 'in' | 'out'
 * @returns {boolean}
 */
function hasAvailablePort(idx, direction) {
  const type = G.grid[idx]?.type;
  if (!type) return false;
  const def = PORT_DEFS[type];
  if (!def) return true; // 未定义的建筑不受限制
  const max = direction === 'in' ? def.maxIn : def.maxOut;
  return getUsedPorts(idx, direction) < max;
}
```

### 4.3 连接拒绝与反馈

当玩家尝试连接一个端口已满的建筑时：

```
场景：玩家拖拽传送带到一台 ATP合成酶(3/3 输出口已满)

1. 视觉反馈：
   - 目标建筑的端口全部闪红色 0.5秒
   - 传送带线变红色虚线（表示无法连接）

2. 文字提示：

3. Toast 弹出：
   "端口已满！此建筑最多连接 3 条输出管线"

4. 不会静默失败 — 必须有明确的否定反馈
```

### 4.4 P1阶段自动传送带的端口处理

P1阶段传送带自动连接，端口限制仍然生效，但处理方式不同：

```
P1 规则：
  1. 自动连接时优先填充空端口
  2. 端口满后不再生成新自动连接
  3. 因为P1只有4种建筑 × 核心上限2台碳源采集器
     实际最大自动连接数很少，端口限制几乎不会被触发
  4. 端口本身在P1不显示（保持新手友好）
     但底层逻辑仍然运行（为P2平滑过渡做准备）

P2+ 规则：
  1. 端口UI可见
  2. 手动连接时检查端口余量
  3. 连接/断开传送带时实时更新端口状态
```

### 4.5 对 `computeBelts()` 的改动

当前 `computeBelts()` 的自动匹配逻辑需要增加端口限制：

```javascript
// 在 computeBelts() 的自动连接部分（P1阶段）
// 原有逻辑：matchFrom/matchTo 抢占式匹配
// 新增约束：
//   1. from端匹配前检查 getUsedPorts(fi, 'out') < PORT_DEFS[type].maxOut
//   2. to端匹配前检查 getUsedPorts(ti, 'in') < PORT_DEFS[type].maxIn
//   3. 抢占时也要检查被抢占者是否有其他可用端口

// 在手动连接入口增加检查：
// manualBeltConnect(fromIdx, toIdx) 中新增：
if (!hasAvailablePort(fromIdx, 'out')) {
  G.log('⚠️ ' + BLDS[G.grid[fromIdx].type].n + ' 输出端口已满', 'w');
  return false;
}
if (!hasAvailablePort(toIdx, 'in')) {
  G.log('⚠️ ' + BLDS[G.grid[toIdx].type].n + ' 输入端口已满', 'w');
  return false;
}
```

### 4.6 对 `getBeltMultiplierForBuilding()` 的影响

端口系统引入后，传送带乘数计算逻辑基本不变——因为端口限制的是连接数，而乘数计算基于已有的有效连接。唯一变化：

```
变化：validBeltCount 的上限从"无限"变为 PORT_DEFS[type].maxIn
效果：不会出现一台建筑被10条传送带供给的极端情况
影响：中后期不会因为过度连接而出现"效率溢出"
```

### 6.2 传送带连接模式中的端口高亮

进入传送带连接模式时（选中起点建筑后）：

```
1. 起点建筑：
   - 输出端口中空闲的 → 亮绿色脉冲（"从这里发出"）
   - 已连的 → 保持原色
   - 全满 → 全红（不能再连了）

2. 所有可连接的目标建筑：
   - 输入端口中空闲的 → 白色脉冲（"可以连到这里"）
   - 端口已满的建筑 → 整体变暗，端口显示红色
   
3. 不在FLOW_MAP中的建筑（不可连接）：
   - 整体变暗，无端口高亮
```

### 6.3 连接建立时的反馈

```
成功连接:
  1. 起点建筑的一个空闲输出端口 → 变为资源色(已连)
  2. 终点建筑的一个空闲输入端口 → 变为资源色(已连)
  3. 传送带线从输出端口圆心连向输入端口圆心（视觉上"从端口出发"）
  4. 音效: 清脆的"咔嗒"声（已有belt_connect音效）

失败（端口满）:
  1. 目标建筑所有端口闪红 0.5s
  2. 短震动（移动端）
  3. log消息 + toast提示
  4. 音效: 低沉的"嗡"声

断开连接:
  1. 对应的输出端口 → 恢复空闲(灰色)
  2. 对应的输入端口 → 恢复空闲(灰色)
  3. 音效: 已有belt_disconnect音效
```

## 九、风险与缓解

| 风险 | 严重性 | 缓解方案 |
|------|--------|---------|
| 手机端端口太小无法辨认 | 高 | 手机端用"端口计数器"替代逐个显示：如 `📥2 📤3` 文字标签 |
| 端口限制让新玩家困惑 | 中 | P1完全隐藏 + P2首次教学面板 + 首次满限制提示 |
| 现有存档兼容性 | 中 | 端口限制仅作用于新连接；已有连接超出端口上限的视为"祖父条款"，不强制断开 |
| 端口数设计不平衡 | 中 | 所有端口数值标记为 `[可调]`，通过调整电子表格迭代 |
| 与培养皿实验区域冲突 | 低 | 培养皿的3×3高亮叠加在端口层之下，不覆盖端口 |
| 增益建筑"无端口"导致玩家以为放错了 | 低 | 增益建筑用特殊光晕标识角色，tooltip明确说明"此建筑无需传送带" |
